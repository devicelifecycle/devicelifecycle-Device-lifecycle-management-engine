Chapter 02

Project Overview — Why We Built This

The business problem, the solution we designed, and the six roles that use it every day.

SCC handles device lifecycle management — companies trade in old phones and laptops, and SCC either resells them, refurbishes them, or processes them through its COE (Center of Excellence) facility. Before this platform existed, all of this was done through spreadsheets, email chains, and manual pricing lookups against competitor websites. That created three major problems:

  • Pricing inconsistency. Different sales reps were quoting different prices for the same device and condition because they were manually checking Bell, Telus, and GoRecell at different times.
  • No triage visibility. When devices arrived at the COE and turned out to be in worse condition than quoted, there was no formal process for handling the discrepancy — just phone calls and spreadsheets.
  • No audit trail. If a customer disputed a price or a condition assessment, there was nothing to point to.

This platform solves all three. It gives every stakeholder — customers, the internal sales team, COE technicians, and managers — a single place where every device's journey is tracked from the moment a quote is requested until the final payment is processed.

The Order Lifecycle

Every device passes through the following state machine. This is enforced at the service layer — no API call can skip a step.

draft → submitted → quoted → accepted → [sourcing → sourced] → shipped_to_coe
→ received → in_triage → qc_complete → ready_to_ship → shipped → delivered → closed

             ↘ rejected (terminal)
             ↘ cancelled (terminal, any stage)

Trade-in orders flow left to right. When devices arrive at the COE in worse condition than quoted, an exception is raised. The customer must approve or dispute the adjusted price before the order can advance past in_triage.

4.2 Request Flow

Every request goes through this chain:

Browser
  → Next.js Middleware (proxy.ts)   ← checks auth cookie, redirects to /login if missing
  → Route Handler (app/api/*)      ← validates JWT, checks role, validates body with Zod
  → Service Class (services/*)     ← business logic, DB queries via Supabase client
  → Supabase (PostgreSQL + RLS)    ← Row Level Security policies filter data per user
  → Service Class                  ← transforms data, applies business rules
  → Route Handler                  ← NextResponse.json(result, { status: 200 })
  → Browser

4.3 Two Supabase Clients

This was one of the most important architectural decisions. There are two different Supabase clients in the codebase, and knowing which to use where prevented several serious bugs:

// src/lib/supabase/server.ts
// Cookie-based — runs as the logged-in user, subject to RLS
export async function createServerSupabaseClient() {
  const cookieStore = await cookies()
  return createServerClient(url, anonKey, { cookies: { ... } })
}
Uses the anon key and the user's session cookie. Supabase enforces RLS policies — a customer can only see their own orders. Used for all user-facing data reads.
// src/lib/supabase/service-role.ts
// Service role key — bypasses RLS entirely
export function createServiceRoleClient() {
  return createClient(url, serviceRoleKey)
}
Uses the service_role key. Bypasses all RLS. Used only after the route handler has already validated the user's permission — for operations that need to touch data across org boundaries, like creating notifications for all admins or running the pricing engine against competitor_prices.

5.2 The Order State Machine

This is the heart of the workflow engine. Defined in src/lib/constants.ts:

export const VALID_ORDER_TRANSITIONS: Record<OrderStatus, OrderStatus[]> = {
  draft:          ['submitted', 'cancelled'],

  quoted:         ['accepted', 'rejected'],
  accepted:       ['sourcing', 'sourced', 'shipped_to_coe', 'cancelled', 'ready_to_ship'],

  sourcing:       ['sourced', 'cancelled'],

  shipped:        ['delivered'],
  delivered:      ['closed'],
  closed:         [],           // Terminal
  cancelled:      [],           // Terminal
}

qc_complete → quoted is a special path for walk-in or unquoted trade-ins where the first quote happens after triage rather than before receiving.

Phase 1 — Core Platform

Feb 3 to Feb 23, 2026. Building the foundation: auth, orders, devices, users, notifications.

6.1 Authentication

Authentication runs on Supabase Auth (GoTrue under the hood). The flow:

  1. User submits email + password to POST /api/auth/login
  2. The route calls supabase.auth.signInWithPassword()
  3. Supabase returns a JWT session — the route stores it in an httpOnly cookie via Supabase's SSR helpers
  4. The Next.js middleware reads that cookie on every request and confirms the session is valid
  5. After login, the route reads the user's role from the users table and redirects to the appropriate dashboard page

The forgot-password flow (added April 2) uses PKCE (Proof Key for Code Exchange) — a security upgrade over the simple link-in-email flow. The user gets a link with a code challenge. When they click it, the reset page exchanges the code for a session before allowing the password change. This prevents link-stealing attacks.

// src/app/api/auth/forgot-password/route.ts
const { error } = await supabase.auth.resetPasswordForEmail(email, {
  redirectTo: `${origin}/reset-password`,
})
Supabase sends the PKCE challenge link to the user's email. The redirectTo must be whitelisted in the Supabase dashboard under Auth → URL Configuration.

6.2 Order Service — Every Method Explained

The OrderService class in src/services/order.service.ts (930 lines) is the central business logic engine. Here is every method:

getOrders(filters)

Fetches orders with pagination, sorting, and role-based scoping. The scoping logic runs before any filters:

if (requester_role === 'customer' && requester_organization_id) {
  // Find all customer records for this org
  const { data: customers } = await supabase
    .from('customers').select('id').eq('organization_id', requester_organization_id)
  // Only return orders belonging to this org's customer accounts
  query = query.in('customer_id', allowedCustomerIds)
}
A customer can only see their own org's orders. This is defense-in-depth — RLS also enforces this at the DB level, but we filter at the service layer too so error messages are clear.
if (search) {
  const safeSearch = sanitizeSearchInput(search)
  // Also search order_items by IMEI/serial — so COE can find an order by device
  const { data: matchingItems } = await supabase
    .from('order_items').select('order_id')
    .or(`imei.ilike.%${safeSearch}%,serial_number.ilike.%${safeSearch}%`)
  if (matchingOrderIds.length > 0) {
    query = query.or(`order_number.ilike.%${safeSearch}%,id.in.(${matchingOrderIds.join(',')})`)
  }
}
Search works on order number AND on IMEI/serial number stored in order_items. A COE tech can type an IMEI to find the order. The sanitizeSearchInput call strips SQL injection characters before the query runs.

createOrder(input, userId, orgId)

Order creation has collision-retry logic because order numbers are generated by a Supabase RPC function:

for (let attempt = 1; attempt <= MAX_ORDER_NUMBER_ATTEMPTS; attempt++) {
  const { data: orderNumber } = await supabase.rpc('generate_order_number', { direction })
  // If RPC returns the same colliding number twice, advance locally
  if (lastConflictedOrderNumber && orderNumber === lastConflictedOrderNumber) {
    orderNumber = await getNextOrderNumberFromTable(supabase, direction)
  }
  const { data, error } = await supabase.from('orders').insert({ order_number: orderNumber, ... })
  if (!error) { order = data; break }
  if (!isOrderNumberConflict(error)) throw error
  lastConflictedOrderNumber = orderNumber
}
Order numbers follow the pattern PO-2026-0001 (inbound) or INV-2026-0001 (outbound). If two requests land simultaneously and generate the same number, the second insert fails with a unique constraint violation (PostgreSQL error code 23505). The retry loop handles this by re-asking for a number or scanning the table to find the next available one.

transitionOrder(id, toStatus, userId, notes)

The state machine enforcer. All status changes go through here:

// Validate the transition is legal
const validTransitions = VALID_ORDER_TRANSITIONS[order.status]
if (!validTransitions.includes(toStatus)) {
  throw new Error(`Invalid transition from ${order.status} to ${toStatus}`)
}
Before touching the database, we check the VALID_ORDER_TRANSITIONS map. This prevents any code path from skipping a required step — e.g. jumping from draft directly to received.
// Auto-generate quote when trade-in order is submitted
if (toStatus === 'submitted' && order.type === 'trade_in') {
  await this.autoQuoteOrderItems(orderId)
}
When a trade-in order is submitted, the pricing engine runs automatically for every item. If it succeeds, the order has prices before any human has to look at it. This is the zero-human-touchpoint flow.
// Record to sales_history when order closes — feeds ML model training
if (toStatus === 'closed') {
  await this.recordSalesHistory(supabase, id, order.type)
}
Every closed order's final prices get written to sales_history. The ML pricing model retrains on this data, so its accuracy improves over time as more deals are completed.

      pricing_metadata: {
        suggested_by_calc: true,
        pricing_source: 'auto',
        confidence: result.confidence,
        anchor_price: result.breakdown?.anchor_price,
      },
    }).eq('id', item.id)
  }
}

6.3 Notification Service

The NotificationService handles three channels: in-app (stored in notifications table, delivered via Supabase Realtime), email (sent via Resend), and SMS (sent via Twilio, only when configured).

static async createNotification(input: {
  user_id: string; type: NotificationType;
  title: string; message: string; link?: string;
  metadata?: Record<string, unknown>
}): Promise<Notification> {
  const supabase = createServiceRoleClient()
  // Always insert to notifications table (in-app)
  const { data } = await supabase.from('notifications').insert({ ... }).select().single()
  // If email type, send via Resend
  if (input.type === 'email' && input.metadata?.email) {
    await EmailService.sendEmail(...)
  }
  return data
}
Every notification — regardless of type — is written to the database. This gives a complete notification history. Email/SMS are fire-and-forget on top. The service role client is used because notifications often need to be created for users in different organizations.

7.2 Supabase Realtime

The order detail page subscribes to changes on the orders table using Supabase Realtime. When any admin updates the order status, all browsers viewing that order see the update within 1-2 seconds without a page refresh:

// In the order detail page component
useEffect(() => {
  const channel = supabase
    .channel(`order:${orderId}`)
    .on('postgres_changes', {
      event: 'UPDATE',
      schema: 'public',
      table: 'orders',
      filter: `id=eq.${orderId}`,
    }, (payload) => {
      // Update local state with fresh order data
      queryClient.invalidateQueries(['order', orderId])
    })
    .subscribe()

  return () => { supabase.removeChannel(channel) }
}, [orderId])

The notification bell also uses a Realtime subscription on the notifications table filtered by user_id. New notifications increment the unread badge without polling.

7.4 Security Hardening

Several important security decisions made during Phase 2:

Security: Service Role Scoping
The createServiceRoleClient() is only used after the route handler has verified the user's role and intent. It is never used to do the permission check itself. This prevents privilege escalation — the service role key bypasses RLS, so it must only be invoked after we've already confirmed the request is authorized.
Security: SQL Injection Prevention
All user-supplied search strings pass through sanitizeSearchInput() which strips SQL-special characters before they reach any .ilike() or .or() call. The Supabase client uses parameterized queries internally, so this is defense-in-depth.
Security: Sort Column Allowlist
The sort_by parameter in all list endpoints is validated against an explicit allowlist before being passed to .order(). Without this, a malicious sort_by=password could expose column names in error messages. 
const ALLOWED_SORT_COLUMNS = ['created_at', 'updated_at', 'order_number', 'status', 'total_amount']
const safeSortBy = ALLOWED_SORT_COLUMNS.includes(sort_by) ? sort_by : 'created_at'
