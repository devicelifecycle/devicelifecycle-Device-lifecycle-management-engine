15.2 Sprint Clusters — What Actually Moved

March 3 to March 12, 2026 — Workflow branching and first stabilization wave

6 active commits across 5 dates
  • March 3: order auto-splitting, vendor bids API, customer privacy boundaries, dark mode, landing page first pass.
  • March 4: cinematic theme layer, shipping fixes, CSV upload templates, security checks moved earlier in pricing routes.
  • March 6: enterprise/banking visual treatment plus general pricing and UI cleanup.
  • March 8: first major competitor-based pricing rewrite, quoting flow refresh, ML training UI.
  • March 12: cleanup day: order persistence bugs, seed corrections, reset-script repair.
Meaning
By mid-March the platform had already stopped being a simple CRUD dashboard. It had split into multiple operational lanes: customer intake, vendor response, pricing intelligence, and workflow visibility.

March 28 to March 31, 2026 — Production-awareness arrives

18 active commits across 3 dates
  • Deployment-aware URL generation fixed production emails that still pointed to localhost.
  • The catalog was expanded and template downloads were made explicit for customer-facing workflows.
  • Vercel cron health, analytics, and timestamp normalization were tightened.
  • Several production 404s were eliminated by aligning front-end calls with real API routes.
  • Auth handoff speed became a first-class concern by March 31.
Why This Matters
This is the point where the project stopped behaving like a local prototype and started reacting to production conditions: URLs, timezone behavior, cron semantics, and missing-route failures.

April 1 to April 5, 2026 — Workflow marathon and presentation hardening

74 active commits across 5 dates
  • April 1: massive workflow repair day. Customer order auth, CSV upload access, exception approval, triage state sync, quote preservation, notification gaps, and login routing all moved at once.
  • April 2: PKCE password reset, MFA, Next.js 16 migration, pricing accuracy and model-comparison endpoints.
  • April 3: mobile-first polish, pricing consistency, duplicate cleanup scripts, stale data fixes, Speed Insights.
  • April 4: receiving/QC/triage enhancements, pricing formula corrections, beat-competitor logic, CSV bulk triage import.
  • April 5: quote summary controls, manual prices, CPO condition cleanup, optimistic triage UI, sign-in speed, cross-device cache busting, order-intake tab.
Pattern
The repeated theme during this window was not adding brand-new features. It was making the already-built workflows demo-safe: fewer dead ends, fewer stale states, fewer role mismatches, clearer quote controls, and faster navigation.

April 6 to April 10, 2026 — Pricing algorithm convergence and receiving realism

57 active commits across 5 dates
  • April 6: market-reference pricing experiment using benchmark tables.
  • April 7: live triage sync, separate PO vs invoice numbering, discrepancy UI, deletion tools, auth speed, realtime.
  • April 8: Bell/Telus/GoRecell policy becomes canonical, Excel triage uploads, fuzzy matching, competitor visibility repair.
  • April 9: diagnostics auto-fill, login overlay and double-navigation fixes, realtime subscriptions extended across plain-fetch pages.
  • April 10: customer auto-submit, exception-service repair, customer dashboard 500 fixes, receiving manifests, mismatch repricing, Vercel deployment repair, vendor and triage hardening.
Architectural Shift
The pricing engine changed several times here because the business rule itself was still being negotiated: first weighted averages, then explicit Bell/Telus averages, then blending with GoRecell, then floors to make sure quotes stayed above GoRecell Fair where required.

April 12 to April 16, 2026 — Parser maturity, scraper integrity, and audit cleanup

60 active commits across 5 dates
  • April 12: smart trade-template import, shared condition normalization, performance work for admin pricing and order creation.
  • April 13: universal parser completion, larger catalog, full design-system pass, 1,331 pricing tests, code splitting, client-wrapper fixes.
  • April 14: GoRecell full-catalog scraper expansion, phantom-device cleanup, variant-safe catalog matching, server-side search, reports rebuild, scraper health, duplicate cleanup.
  • April 15: stale-data warnings, SLA auto-escalation, scrapling pilot routing, AI-assisted exception suggestions, brand margin overrides, manual-only shipping.
  • April 16: docs deliverables plus audit fixes to pricing precedence, triage resolution semantics, and exception queue behavior.
Outcome
By April 16, the project had shifted from broad feature building to high-precision correction work: bad scraper mappings, stale competitor data, type edge cases, audit expectations, and explanation/documentation quality.

A. Pricing blend logic in src/services/pricing.service.ts

function buildApprovedTradeInPricingBlend(
  list: Array<{ name: string; price: number }>
) {
  const deduped = new Map()
  for (const entry of list) {
    if (!Number.isFinite(entry.price) || entry.price <= 0) continue
    const canonical = getApprovedTradeInPricingCompetitorName(entry.name)
    if (!canonical) continue
    if (!deduped.has(canonical)) deduped.set(canonical, { name: canonical, price: entry.price })
  }

  const approvedCompetitors = Array.from(deduped.values())
  const bellTelusCompetitors = approvedCompetitors.filter(
    (entry) => entry.name === 'Bell' || entry.name === 'Telus'
  )
  const bellTelusAvg = bellTelusCompetitors.length > 0
    ? bellTelusCompetitors.reduce((sum, entry) => sum + entry.price, 0) / bellTelusCompetitors.length
    : 0
  const goRecellPrice = approvedCompetitors.find((entry) => entry.name === 'GoRecell')?.price ?? 0

  let referencePrice = 0
  if (bellTelusAvg > 0 && goRecellPrice > 0) referencePrice = (bellTelusAvg + goRecellPrice) / 2
  else if (goRecellPrice > 0) referencePrice = goRecellPrice
  else if (bellTelusAvg > 0) referencePrice = bellTelusAvg
}
What each block does: the loop first removes bad numeric rows, then canonicalizes names so spelling or casing differences collapse into the three approved competitors. Bell and Telus are averaged together first because they represent the carrier reference market. GoRecell is then blended with that carrier average to produce the working reference price. The fallback branches avoid returning zero when only one side is available.

B. Header detection in src/app/api/orders/parse-trade-template/route.ts

function findHeaderRow(rawRows: unknown[][]): { headerIdx: number; groupIdx: number | null } {
  let bestScore = 0
  let headerIdx = 0
  const limit = Math.min(rawRows.length, 15)
  for (let i = 0; i < limit; i++) {
    const score = scoreHeaderRow(rawRows[i])
    if (score > bestScore) { bestScore = score; headerIdx = i }
  }
  const groupIdx = (headerIdx > 0 && rawRows[headerIdx - 1].filter(c => String(c ?? '').trim()).length >= 2)
    ? headerIdx - 1
    : null
  return { headerIdx, groupIdx }
}
Why this mattered: many customer spreadsheets do not start with headers on row 0. This function scans the first 15 rows, scores them against known header patterns, and optionally detects a group-label row above the real headers. Without this step, the parser would treat company names, date rows, or merged title rows as data.

C. Safe catalog matching in src/lib/scrapers/pipeline.ts

export function isSafeCatalogModelMatch(scrapedModel: string, catalogModel: string): boolean {
  if (scrapedModel === catalogModel) return true

  const scrapedExtendsCatalog = modelTokenMatch(scrapedModel, catalogModel) && scrapedModel !== catalogModel
  const catalogExtendsScraped = modelTokenMatch(catalogModel, scrapedModel) && catalogModel !== scrapedModel

  if (!scrapedExtendsCatalog || catalogExtendsScraped) return false

  return hasCompatibleVariantProfile(scrapedModel, catalogModel)
}
Root cause addressed: earlier scraper runs allowed longer scraped names like iPhone 14 Pro Max to land on shorter catalog rows like iPhone 14. This guard forces directionality and compatible variant keywords so Pro, Max, Plus, Ultra, and similar suffixes do not silently collapse into the wrong device.

17.1 The deepest risk area: pricing + scrapers + catalog

The most sensitive subsystem was the combination of competitor scraping, device matching, and final quote generation. A minor bug in any one layer could look like a pricing bug in the UI even when the UI was rendering correctly. That is why so many late-stage fixes touched both src/lib/scrapers/pipeline.ts and src/services/pricing.service.ts together.

Audit Takeaway
If a future quote looks wrong, inspect the stack in this order: saved competitor rows, scraper model/storage normalization, approved-competitor selection, staleness filters, pricing settings, then UI display. Skipping straight to the UI is usually the wrong debugging order.

17.2 The second deepest risk area: auth + routing + realtime

The next recurring cluster was auth/state propagation. The app mixes Supabase auth, role-aware dashboards, client-side transitions, and realtime updates. Bugs here were rarely “bad password” problems. They were usually timing problems: cached user too early, navigation before session hydration, stale subscription state, or multiple fetches racing each other.
