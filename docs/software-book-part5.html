Its real job is not rendering. It is mediation. The page mediates between:

  • manual shipping creation and tracking visibility

The creation pages needed strong template support not because users dislike forms, but because enterprise customers already have spreadsheets. The product had to adapt to their workflow rather than force them into a new one.

29.2 vendor.service.ts

The vendor service handles vendor-side economics and allocation. It supports direct assignment, open bidding, auto-accept for full-quantity first bids, and auto-splitting when accepted bids jointly cover the whole order.

static async submitBid(input: {...}): Promise<VendorBid & { auto_accepted?: boolean }> {
  const bid = await supabase.from('vendor_bids').insert({...status:'pending'}).select().single()
  ...
  if (order && !order.vendor_id && input.quantity >= (order.total_quantity || 0)) {
    const accepted = await this.updateBidStatus(bid.id, 'accepted')
    return { ...accepted, auto_accepted: true }
  }
  return bid
}
This is one of the workflow accelerators in the system: the first complete-coverage bid can be auto-accepted, reducing manual admin work for straightforward sourcing cases.

29.3 shipment.service.ts

The shipment service reflects a key product simplification: shipping is now fundamentally manual-tracking first. It still tracks statuses, webhooks, receiving, and notifications, but the system no longer depends on in-app label purchasing.
